{/* # Maîtriser l’art du prompt pour les développeurs */}

Avec l’arrivée massive des **LLM (Large Language Models)** dans les workflows de développement, savoir **écrire un bon prompt** est devenu une compétence à part entière.

Un prompt mal formulé donne des réponses vagues, imprécises ou inutilisables. À l’inverse, un prompt bien conçu peut transformer une IA en **pair programmer**, **architecte**, **reviewer** ou **assistant technique** fiable.

Dans cet article, nous allons voir comment concevoir des prompts **robustes, précis et exploitables**, spécifiquement adaptés aux besoins des développeurs.

---

## Pourquoi le prompt engineering est une compétence de développeur

Un prompt n’est **pas une simple question**, mais une **spécification fonctionnelle**.

Comme pour du code :

* prompt flou → comportement imprévisible
* prompt précis → résultat cohérent
* prompt structuré → sortie exploitable

Le prompt engineering se rapproche davantage de la **conception d’API** que d’une conversation.

---

## Anatomie d’un bon prompt

Un prompt efficace contient généralement cinq blocs :

1. Le contexte
2. Le rôle attendu
3. L’objectif
4. Les contraintes
5. Le format de sortie

---

## 1. Fournir le contexte

Sans contexte, le modèle doit faire des hypothèses.

❌ Exemple vague :

> Explique ce code

✅ Exemple précis :

> Voici un composant React Server Component utilisé dans une application Next.js App Router. Explique son fonctionnement et ses implications côté serveur.

---

## 2. Définir un rôle explicite

Définir un rôle permet d’orienter le raisonnement.

```text
Tu es un développeur senior spécialisé en TypeScript et architecture frontend.
```

Exemples de rôles utiles :

* développeur senior
* architecte logiciel
* reviewer de code
* expert sécurité
* mentor pédagogique

---

## 3. Définir un objectif clair

Un prompt doit répondre à **une intention principale**.

❌ Trop vague :

> Améliore ce code

✅ Clair :

> Propose une version plus lisible et plus performante de ce code en respectant les principes SOLID.

---

## 4. Ajouter des contraintes techniques

Les contraintes transforment un prompt généraliste en prompt exploitable.

Exemples de contraintes :

* TypeScript strict
* Next.js App Router
* pas de dépendance externe
* compatible Node.js 18+
* approche fonctionnelle

```text
Contraintes :
- TypeScript strict
- Aucun package externe
- Compatible React Server Components
```

---

## 5. Forcer un format de sortie

Sans format, la réponse est souvent difficile à exploiter.

Formats utiles :

* liste numérotée
* tableau comparatif
* code uniquement
* checklist
* étapes séquentielles

```text
Réponds avec :
1. Une explication courte
2. Un exemple de code
3. Les pièges à éviter
```

---

## Exemple de prompt complet orienté développement

```text
Tu es un développeur senior spécialisé en TypeScript et Next.js.

Contexte :
Je développe un blog avec Next.js App Router et des fichiers MDX.

Objectif :
Concevoir une fonction serveur pour parser, valider et trier des articles MDX.

Contraintes :
- TypeScript strict
- Validation avec Zod
- Aucun accès client
- Compatible avec la génération statique

Format de sortie :
- Explication concise
- Code complet
- Liste des erreurs courantes
```

## Prompts courants pour développeurs

### Génération de code

```text
Génère une fonction TypeScript pure.
Inclure des tests unitaires.
Aucune dépendance externe.
```

### Review de code

```text
Analyse ce code comme un reviewer senior.
Identifie :
- problèmes de lisibilité
- risques de bugs
- améliorations possibles
```

### Debugging

```text
Voici un bug reproductible.
Explique :
1. La cause probable
2. La correction
3. Comment éviter ce bug
```

### Architecture logicielle

```text
Propose une architecture scalable pour une application SaaS.
Stack : Next.js, PostgreSQL, Auth.
Inclure un schéma logique.
```

---

## Anti-patterns fréquents

* Prompts trop longs sans structure
* Objectifs multiples contradictoires
* Aucune contrainte technique
* Demander "le meilleur" sans critères
* Mélanger frontend, backend et infra

---

## Itération et amélioration continue

Un prompt est rarement parfait du premier coup.

Méthode recommandée :

1. Prompt simple
2. Analyse de la réponse
3. Ajout de contraintes
4. Raffinement du format
5. Sauvegarde du prompt final

Un prompt se **refactorise** comme du code.

---

## Organiser ses prompts comme du code

Bonnes pratiques :

* stocker dans des fichiers `.md`
* versionner avec Git
* nommer clairement les prompts
* documenter leur objectif

---

## Le prompt comme outil de développement

À terme, les prompts deviennent :

* des templates
* des outils internes
* des accélérateurs de productivité
* une extension naturelle du code

Un bon développeur ne se contente plus d’écrire du code : il **conçoit des interfaces avec l’IA**.

---

## Conclusion

Le prompt engineering est une compétence technique à part entière.

À retenir :

* Un prompt est une spécification
* Le contexte est crucial
* Les contraintes font la qualité
* Le format change tout
* L’itération est indispensable
