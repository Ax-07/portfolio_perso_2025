{/* # Utiliser GitHub Copilot efficacement dans VS Code et les IDE modernes */}

GitHub Copilot est bien plus qu’un simple outil d’autocomplétion. Utilisé correctement, il devient un **assistant de développement contextuel**, capable de comprendre ton projet, tes conventions et tes intentions.

Pourtant, dans beaucoup de projets, Copilot est utilisé de manière passive : suggestions acceptées au hasard, résultats incohérents, ou code généré difficile à maintenir.

Dans cet article, nous allons voir **comment utiliser Copilot de manière structurée et professionnelle**, en particulier grâce au fichier `copilot-instructions.md` et à l’organisation d’un dossier `instructions`.

---

## Comment fonctionne réellement GitHub Copilot

Copilot génère ses suggestions à partir de :

* le fichier courant
* les fichiers ouverts récemment
* la structure du projet
* les commentaires et instructions textuelles
* certains fichiers de contexte spécifiques

👉 Plus le contexte est clair, plus les suggestions sont pertinentes.

Copilot ne "devine" pas : il **interprète ce que tu lui donnes**.

---

## Le problème : un contexte implicite et instable

Sans cadre explicite, Copilot peut :

* mélanger les styles de code
* ignorer les conventions du projet
* proposer des patterns inadaptés
* générer du code difficile à maintenir

La solution consiste à **externaliser les règles du projet** dans des fichiers dédiés.

---

## Le rôle clé de `copilot-instructions.md`

Le fichier `copilot-instructions.md` permet de définir des **instructions globales** que Copilot doit suivre dans tout le projet.

Il agit comme une **charte de développement** lue implicitement par Copilot.

### Emplacement recommandé

À la racine du projet, dans le dossier `.github` (recommandé par la doc) :

```
/.github/copilot-instructions.md
```

---

## Exemple de `copilot-instructions.md`

```md
# GitHub Copilot – Project Instructions

## Général
- Langage principal : TypeScript
- Style : clair, explicite, maintenable
- Privilégier les fonctions pures
- Éviter la logique implicite

## Frontend
- Framework : Next.js App Router
- Utiliser les React Server Components par défaut
- Pas de logique métier dans les composants UI

## Backend
- Validation systématique avec Zod
- Pas d'accès direct aux variables d'environnement côté client

## Qualité
- Code lisible avant d'être optimisé
- Ajouter des commentaires uniquement si nécessaire
- Préférer la clarté à la concision
```

👉 Ce fichier sert de **boussole** à Copilot.

---

## Le dossier `.github/instructions` : instructions ciblées par chemin (path-specific)

Au-delà des règles globales, il est utile de créer un dossier dédié pour des **instructions spécifiques par contexte**.

### Structure recommandée

```
/.github
  ├── copilot-instructions.md
  └── instructions
      ├── frontend.instructions.md
      ├── backend.instructions.md
      ├── tests.instructions.md
      └── architecture.instructions.md
```

📌 Important : les fichiers doivent **se terminer par** `.instructions.md`.

Ces fichiers servent de **prompts persistants**, réutilisables et versionnés.

---

## Exemple : `.github/instructions/frontend.instructions.md`

```md
---
applyTo: "**/*.{ts,tsx}"
---

# Instructions Frontend

- Utiliser TypeScript strict
- Composants fonctionnels uniquement
- Pas de logique métier dans les composants
- Hooks personnalisés pour la logique complexe
- Préférer la composition à l’héritage
```

Lorsque Copilot travaille sur un fichier qui **match** le `applyTo`, il peut automatiquement prendre en compte ces instructions (en plus des instructions globales). Évite les contradictions : en cas de conflit, le comportement peut devenir non-déterministe.

---

## Exemple : `.github/instructions/tests.instructions.md`

```md
---
applyTo: "**/*.test.{ts,tsx}"
---

# Instructions Tests

- Framework : Vitest
- Tester le comportement, pas l’implémentation
- Nommer les tests de manière explicite
- Un test = une responsabilité
```

---

## Comment "parler" à Copilot (dans le code et dans le chat)

Copilot réagit très bien aux commentaires bien formulés.

### Exemple efficace

```ts
// Objectif : parser un article MDX et valider son front matter avec Zod
// Contraintes : TypeScript strict, aucune dépendance externe
```

👉 Ces commentaires servent de **micro-prompts**. Dans Copilot Chat, tu peux aussi coller un extrait de `copilot-instructions.md` si tu veux forcer un rappel ponctuel.

---

## Bonnes pratiques d’utilisation quotidienne (VS Code / JetBrains / autres)

* Écrire des commentaires d’intention avant le code
* Ne pas accepter aveuglément les suggestions
* Lire chaque proposition comme une PR
* Supprimer le code inutile généré
* Corriger le style immédiatement

Copilot est un **assistant**, pas un décideur.

---

## Anti-patterns courants

* Faire confiance à Copilot sans relecture
* Laisser Copilot définir l’architecture
* Mélanger plusieurs styles de code
* Ignorer les instructions du projet

---

## Copilot comme outil d’équipe

Les fichiers `copilot-instructions.md` et `instructions/` permettent :

* d’harmoniser le code
* de réduire les divergences de style
* d’onboarder plus vite les nouveaux développeurs
* de documenter implicitement les conventions

Copilot devient alors un **outil collectif**, pas individuel.

---

## Conclusion

GitHub Copilot est puissant, mais seulement si on lui fournit un cadre clair.

À retenir :

* Copilot dépend du contexte
* `copilot-instructions.md` définit les règles globales
* le dossier `instructions` apporte du contexte ciblé
* les commentaires sont des prompts
* le développeur reste responsable du code

Utilisé correctement, Copilot devient un **véritable accélérateur de qualité et de productivité**.
